**Change 2 — Replace gallery img tags:**

Replace the gallery section (currently lines 48-52):

```astro
<div class="project-gallery">
  {data.images.map((src) => (
    <img src={src} alt="" loading="lazy" />
  ))}
</div>
```

With:

```astro
<div class="project-gallery">
  {data.images.map((src) => (
    <Image
      src={src}
      alt=""
      width={1600}
      height={1200}
      format="webp"
      widths={[640, 960, 1280, 1600]}
      sizes="(max-width: 640px) calc(100vw - 32px), calc(100vw - 64px)"
      loading="lazy"
    />
  ))}
</div>
```

Width 1600 and height 1200 are reasonable defaults for portfolio gallery images at 4:3 ratio. The `widths` array generates a responsive srcset with four breakpoints. The `sizes` attribute tells the browser: on mobile (max-width 640px) the image fills viewport minus 32px (2x 16px margin), on desktop it fills viewport minus 64px (2x 32px margin) — matching the existing `.project-gallery { margin-inline: var(--space-xl) }` and mobile `var(--space-md)` styles.

The `format="webp"` attribute requests WebP output from Astro's image pipeline.

NOTE: With string paths from /public/, Astro will render these as standard img tags with the specified width/height (preventing CLS) but will NOT perform WebP conversion or srcset generation. This is a known Astro limitation for public folder images. The component pattern is correct and will deliver full optimization once actual images are placed in src/assets/ and the schema migrates to Astro's image() helper. The <Image> component scaffolding is the right architectural step regardless.

Do NOT change the cover image (`<img src={data.cover}` on line 24) — it uses `loading="eager"` and is above the fold.

Do NOT change any CSS in the `<style>` block. The existing `.project-gallery img` styles will apply to the rendered `<img>` element output by `<Image>`.
