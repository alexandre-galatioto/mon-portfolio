**Exception — image container max-width:** A `max-width: 1280px` constraint is added to
image containers (project gallery, home page grid, cover image, about portrait) and centered
with `margin-inline: auto`. This is not a new spacing token — it is a layout constraint
applied directly as a pixel value. Chosen at 1280px (within CONTEXT.md D-09's 1200–1400px
range) to match the widths declared in the Astro `<Image>` `widths` attribute: `[640, 960,
1280, 1600]` — 1280px ensures sharp rendering at the largest commonly used desktop viewport
without overshooting into the 1600px fallback tier.

**Source:** CONTEXT.md D-09 (max-width cap authorized); Claude's discretion (exact 1280px value)

## Typography

All tokens carry forward from Phase 2 unchanged. No new sizes, weights, or roles in Phase 3.

| Role | Size | Weight | CSS Weight | Line Height |
|------|------|--------|------------|-------------|
| Body | 16px `--font-size-base` | Regular | 400 | 1.5 |
| Label | 14px `--font-size-sm` | Regular | 400 | 1.4 |
| Heading | 24px `--font-size-lg` | Bold | 700 | 1.2 |
| Display | 40px `--font-size-xl` | Bold | 700 | 1.1 |

**Font loading change in Phase 3:**

The Regular (400) weight `@font-face` already uses `font-display: swap` (already present in
`global.css` for all four weights). Phase 3 adds a `<link rel="preload">` tag in `BaseLayout.astro`
for the Regular weight only:

```html
<link
  rel="preload"
  href="/fonts/NeueHaasUnica-Regular.woff2"
  as="font"
  type="font/woff2"
  crossorigin
/>
```

Medium (500), Bold (700), and Italic (400i) weights load on demand via `font-display: swap`
with no preload — they are not on the critical render path.

**Source:** CONTEXT.md D-06; Phase 1 UI-SPEC (font-display: swap already declared)

## Responsive Contract

This section is the primary new content in Phase 3's UI-SPEC. It audits the existing
responsive implementation and specifies what must be verified and added.

### Breakpoints in Use

| Breakpoint | Value | Governs |
|------------|-------|---------|
| Mobile | `max-width: 640px` | Page layouts (home grid, project meta, about layout) |
| Nav collapse | `max-width: 768px` | Navigation hamburger / overlay |

**Source:** CONTEXT.md D-10 (existing breakpoints kept as-is); Phase 1 UI-SPEC (768px nav)

### Verified Responsive Layouts

The following layouts already have `@media (max-width: 640px)` rules in their page files
and must be verified to render correctly at actual mobile viewport widths:

| Page / Component | Desktop Layout | Mobile Layout (< 640px) | Verify |
|-----------------|----------------|--------------------------|--------|
| Home page grid | `grid-template-columns: repeat(2, 1fr)`, `padding-inline: 32px` | `grid-template-columns: 1fr`, `padding-inline: 16px` | Images fill full column width, 3:2 aspect ratio preserved |
| Project meta section | `grid-template-columns: 1fr 1fr`, `padding-inline: 32px` | `grid-template-columns: 1fr`, `padding-inline: 16px` | Left column (title, type) stacks above right column (description, link) |
| Project cover image | `margin-inline: 32px`, `height: 50vh` | `margin-inline: 16px` | `50vh` height preserved on mobile, no distortion |
| Project gallery | `margin-inline: 32px`, `gap: 32px` | `margin-inline: 16px` | Images fill container width, no horizontal scroll |
| Prev/next nav | `padding-inline: 32px` | `padding-inline: 16px` | Titles readable, no overflow |
| About layout | `grid-template-columns: 1fr 1fr`, `padding-inline: 32px` | `grid-template-columns: 1fr`, `padding-inline: 16px` | Portrait stacks above content |
| Navigation | Desktop links visible, hamburger hidden | Desktop links hidden, hamburger visible | Overlay opens/closes correctly |

### Tablet Behavior (640px – 1024px)

No tablet-specific breakpoint is required. Layouts transition smoothly from 2-column (>640px)
to single-column (<=640px). The 2-column grid at 641px–1024px is acceptable — content remains
readable at tablet widths. No intermediate breakpoint is added.

**Source:** CONTEXT.md D-10 (discretion delegated); Claude's discretion (no intermediate breakpoint needed)

### Image Fill Behavior

| Context | Behavior | CSS |
|---------|----------|-----|
| Home grid images | `aspect-ratio: 3/2`, `object-fit: cover`, fills column width | Existing — verify only |
| Project cover | `width: 100%`, `height: 50vh`, `object-fit: cover` | Existing — verify only |
| Project gallery images | `width: 100%`, natural height (no crop), preserves original aspect ratio | D-08: no cropping on gallery. Remove any fixed `height` constraint if present |
| About portrait | `width: 100%`, natural height, `object-fit: cover` | Existing — verify only |

**Gallery images must NOT have a fixed height or aspect-ratio constraint** — they display at
natural proportions to honor the art director's original compositions (CONTEXT.md D-08).
The `<Image>` component already uses `width={1600} height={1200}` as generation hints only;
the rendered `<img>` must not have `height` forced via CSS.

**Source:** CONTEXT.md D-08, D-09 (image fill behavior and max-width cap)

### Image Container Max-Width

Apply to all full-width image containers and the home page grid:

```css
max-width: 1280px;
margin-inline: auto;
```

Applied to: `.project-grid`, `.project-cover`, `.project-gallery`, `.about-portrait`.

On viewports wider than ~1344px (1280px + 2 × 32px margin), the content area stops growing
and centers on the canvas. Below that threshold, existing `margin-inline` padding applies
as normal.

**Implementation note:** The `max-width` wraps the container, not individual images. Existing
`margin-inline` padding tokens remain on the inner elements and continue to function.

**Source:** CONTEXT.md D-09; Claude's discretion (1280px value)

## Performance Contract

### Image Optimization — Migration from `public/` to `src/assets/`

**What changes:** Project images currently referenced from `public/` are moved to
`src/assets/`. This activates Astro's build-time image pipeline (WebP generation,
responsive srcset). The `<Image>` component import in `[slug].astro` already uses the
correct API — only the image file location and front matter path strings change.

**Visual impact:** None visible to user. Images continue to display identically. Render
behavior is unchanged; only the delivery format (WebP vs original) and srcset improve.

**`sizes` attribute values (carry-forward from Phase 2):**

```
sizes="(max-width: 640px) calc(100vw - 32px), calc(100vw - 64px)"
```

With the 1280px max-width cap, update `sizes` on gallery images to:

```
sizes="(max-width: 640px) calc(100vw - 32px), min(calc(100vw - 64px), 1280px)"
```

This prevents the browser from fetching larger variants than the 1280px cap will ever display.

**Source:** CONTEXT.md D-05; Phase 2 UI-SPEC (Image component usage)

### Font Preload

Add to `<head>` in `src/layouts/BaseLayout.astro` before any stylesheet link:

```html
<link
  rel="preload"
  href="/fonts/NeueHaasUnica-Regular.woff2"
  as="font"
  type="font/woff2"
  crossorigin
/>
```

Only the Regular (400) weight is preloaded. All other weights use `font-display: swap`
without preload — they are not on the critical render path.

**Source:** CONTEXT.md D-06

### Lighthouse Target

| Metric | Target |
|--------|--------|
| Performance score | ≥ 90 |
| Page load time | < 2 seconds on standard connection |
| CLS (Cumulative Layout Shift) | < 0.1 — achieved by declaring explicit `width` and `height` on `<Image>` component calls |
| LCP (Largest Contentful Paint) | < 2.5s — home cover images eager-loaded or preloaded |

**LCP optimization:** The first visible image on the home page (project[0] cover) should use
`loading="eager"` rather than `loading="lazy"`. All subsequent images keep `loading="lazy"`.
This is an HTML attribute change only — no CSS or visual change.

**Source:** CONTEXT.md D-07; REQUIREMENTS.md PERF-01

## Deployment Configuration

No visual components are added for deployment. The contract covers configuration artifacts only.

### `netlify.toml`

Add to project root. No visual impact.

```toml
[build]
  command = "npm run build"
  publish = "dist/"

[[headers]]
  for = "/fonts/*"
  [headers.values]
    Cache-Control = "public, max-age=31536000, immutable"

[[headers]]
  for = "/*.html"
  [headers.values]
    Cache-Control = "public, max-age=0, must-revalidate"
```

Font files are immutably cached (1 year) since filenames are content-stable. HTML files are
revalidated on each deploy. This is the only server-side behavior added.

**Source:** CONTEXT.md D-01, D-02; Claude's discretion (cache headers)
